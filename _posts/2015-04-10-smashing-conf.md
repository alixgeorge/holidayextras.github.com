## Making a silky smooth web - Paul Lewis [@aerotwist](http://twitter.com/aerotwist)

* Layout, painting, composite [csstriggers.com](http://csstriggers.com)
* Tools not rules
* `requestAnimationFrame` > `setTimeout`
* Use workers for lonnng JS
* Devtools timeline JS profiler
* CSS selectors
  - Did I limit the scope enough?
  - Are the selectors fast?
* Layout
  - Forced synchronous layout
  - Layout thrashing
  - FastDOM - batch up reads and writes
* Paint
  - Devtools timeline - paint profiler
* Reducing paint layer
  - `will-change`
  - `translateZ(0)` new layer
* Don't promote all the things, layers use memory
* Reduce paint complexity, look at the tools
* Composite
  - transform & opacity when on own layer
  - self-time, aim for ~5ms
* [RAIL](http://bit.ly/rail-perf)
  - Response (100ms after touch to respond)
  - Animation (10ms if we can)
  - Idle (moar work!?) 50ms at a time
  - Load (1000ms aim)

## UX of offline-first - Jake Archibald [@jaffathecake](http://twitter.com/jaffathecake)

* [SVGOMG](https://jakearchibald.github.io/svgomg/)
* Let users play level one as soon as level one is ready
* Use markup for markup!
* Use _actual_ content
* Unblock your scripts
* Lie about performance
* Don't make fonts required on first render
* [React]([React]), [Ember fastboot](https://github.com/tildeio/ember-cli-fastboot)
* Critical CSS
* Distract the user with an actual experience
* Load scripts separately if functionality is different? HTTP2 benefits?
* Render time punished by assets loading quickly (`requestAnimationFrame`)
* The time people hate wasting the most is right now
* Lie-fi is like offline but it trolls you by pretending to be online
* Treat the network as an enhancement
* Service workers
  - Unobtrusive updates
  - "Update ready/available"
* Users don't trust offline web, _yet_
